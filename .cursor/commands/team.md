# /team

> **注意**：此文件是全局备份。权威版本位于 `common/workspace-template/.cursor/commands/team.md`

**人只需要说"做一个 xxx"，AI 自动完成全部开发工作。**

## 流程

```
人: /team 做一个用户认证
AI: → 快速实现 → TDD验证 → 审查发现问题 → 修复 → 再次验证 → E2E → 归档
```

## 核心理念

**后置发现问题，而非前置避免问题。**

| 旧模式 | 新模式 |
|--------|--------|
| 设计阶段解决所有问题 | 快速迭代，发现问题再修复 |
| 一次性完成 | 小步快跑，多次验证 |
| 预防为主 | 验证为主 |

## 核心能力

| 工具 | 用途 |
|------|------|
| **OpenSpec** | 需求规范 + 进度管理（边做边记） |
| **TDD** | 代码质量保障 |
| **Code Review** | 代码审查 |

## Agent 编排

- `@tdd-guide` - 快速实现 + TDD 验证（核心）
- `@code-reviewer` - 审查发现问题
- `@openspec` - 边做边记，记录规范和进度

## 工作流

### 1. 快速实现 + TDD

```
@tdd-guide 实现用户登录
→ 先写最小可行代码
→ 写测试 → 测试失败 = 发现需求问题 → 问用户
→ 测试通过 = 需求理解正确
→ 不要过度设计
```

### 2. 审查发现问题

```
@code-reviewer 审查代码
→ 发现问题 → 分类: 严重/中等/低
→ 严重问题必须修复
```

### 3. 修复循环

```
@tdd-guide 修复问题
→ 再次审查
→ 直到所有严重问题解决
```

### 4. E2E 验证

```
→ 运行完整测试
→ 发现集成问题 → 修复
→ 直到全部通过
```

### 5. 边做边记

```
→ 完成后更新 spec.md 反映最终实现
→ 更新 tasks.md 标记完成
```

## 最佳实践

1. **快速迭代**：先完成，再完美
2. **测试驱动**：测试失败 = 发现需求问题
3. **持续验证**：多次审查，多次修复
4. **边做边记**：完成后记录规范
5. **问题分类**：严重问题必须修复，低问题可忽略
