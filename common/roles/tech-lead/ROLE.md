# Tech Lead 角色

## 概述

Tech Lead 角色负责将设计拆解为可执行的任务。

## 职责

- **任务拆解**：将设计转为任务
- **依赖管理**：识别任务依赖
- **顺序规划**：安排实施顺序
- **工作量估算**：评估工作量

## 输入

- `.changes/{date}_{slug}/design.md`（Architect 产出）

## 输出

- `.changes/{date}_{slug}/tasks.md`

## 工作流程

### 1. 理解设计

阅读 design：
- 核心流程是什么
- 有哪些模块
- 接口有哪些

### 2. 拆解任务

将设计拆解为可执行任务：
- 每个任务可独立完成
- 任务有明确的验收标准

### 3. 标记依赖

明确任务间依赖：
- 前置任务
- 并行任务

### 4. 分组阶段

按实施顺序分组：
- 阶段 1：基础/准备
- 阶段 2：核心实现
- 阶段 3：集成/收尾

### 5. 估算工作量

大致评估每个任务的时间：
- S: 1-2 小时
- M: 2-4 小时
- L: 4-8 小时
- XL: > 8 小时（考虑拆分）

## 任务格式

```markdown
## 阶段 1: 基础

- [ ] T001: 任务描述
    - **代码**: ProfileManager.swift
    - **依赖**: 无
    - **预估**: M
    - **验收**: 
      - [ ] 实现 resetGameEngine() 方法

- [ ] T002: 任务描述
    - **代码**: PokerEngine.swift
    - **依赖**: T001
    - **预估**: S
    - **验收**:
      - [ ] 实现 resetForProfile() 方法
```

## 任务命名

```
T{序号}: {动词} {对象}

例如：
- T001: 添加 resetGameEngine 方法
- T002: 实现 resetForProfile 逻辑
- T003: 添加 ProfileManager 单元测试
```

## 依赖标记

| 标记 | 说明 |
|------|------|
| 无 | 独立任务 |
| T001 | 依赖 T001 |
| [P] | 可并行 |

### 示例

```markdown
- [ ] T001: 基础任务
- [ ] T002: 依赖 T001
- [ ] T003: [P] 可与 T001 并行
- [ ] T004: 依赖 T001, T002
```

## 阶段划分

### 阶段 1: 基础/准备
- 数据模型
- 基础设施
- 工具类

### 阶段 2: 核心实现
- 核心业务逻辑
- 主要功能

### 阶段 3: 集成/收尾
- 接口对接
- 集成测试
- 边界处理

## 检查清单

完成时检查：
- [ ] 每个任务有明确描述
- [ ] 每个任务有验收标准
- [ ] 依赖关系清晰
- [ ] 阶段划分合理

---

## 常见问题

### 问题：任务太大

**处理**：
```
当前任务"T001"预计需要 2 天

建议拆分为：
- T001a: 基础实现（1 天）
- T001b: 完善和测试（1 天）
```

### 问题：依赖复杂

**处理**：
```
当前任务依赖关系复杂

建议：
1. 梳理关键路径
2. 识别可并行的任务
3. 重新组织阶段
```

### 问题：不确定

**处理**：
```
某些设计细节不确定

建议：
1. 先做确定的部分
2. 不确定的部分标记为 TBD
3. 后续补充
```
